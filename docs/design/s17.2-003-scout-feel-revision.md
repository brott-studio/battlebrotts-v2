# [S17.2-003] Spec revision: scout-feel write-site categorization

**Author:** Gizmo (design)
**Sprint:** S17.2 (S17 Eve Polish Arc)
**Status:** Supplements `docs/design/s17.2-scout-feel.md`. Does NOT replace it. All scope-gate language, invariants, and §4 smoothing math in the original spec stand unless explicitly overridden here.
**Trigger:** Nutts's S17.2-003 impl attempt surfaced a TENSION-retreat / backup-budget interaction the original spec's §3 write-site audit did not trace. This doc closes that gap.

---

## 1. Context

### 1.1 What the original spec covered

- Root cause: `b.position += direction * speed` with a fresh direction every tick, zero angular inertia on the velocity vector, direction and magnitude recomputed from geometry every tick.
- Proposal: add `b.velocity: Vector2` as first-class state; add `_smooth_velocity(b, desired, dt)` helper that (a) rotates `b.velocity` toward desired direction at a per-chassis `max_angular_velocity` cap, (b) blends magnitude at existing chassis `accel`/`decel`, (c) applies reversal damping on flips > 120° for 2 ticks at 0.35×.
- §5 pattern: "collapse the multiple `b.position +=` calls into one `desired_vel` accumulator per tick per bot, then apply `_smooth_velocity` once." One accumulator, one smooth call.
- §10 already flagged two known bypass sites: unstick nudge (S17.2-002's `_apply_unstick_nudge`) and a default-bypass recommendation for separation push. Those two calls were the only exemptions named.

### 1.2 What Nutts's impl attempt uncovered

TENSION orbit-phase in `_do_combat_movement` has **backward retreat logic** when the bot is inside the engagement tolerance band:

    if dist < ideal - tolerance:
        if b.backup_distance < TILE_SIZE:
            var step: float = minf(orbit_spd, TILE_SIZE - b.backup_distance)
            b.position -= to_target.normalized() * step
            b.backup_distance += step

Baseline (direct position writes): each tick consumes the `backup_distance` budget in lockstep with the commanded step. Budget caps at one tile (32 px) per phase, so total backward travel per TENSION entry is bounded by 32 px.

Velocity-smoothing wrapper (naive application of original §5 "one accumulator, one smooth call"): the backward `step` feeds the `desired_vel` accumulator, gets rotated/magnitude-blended through `_smooth_velocity`, and the **realized** displacement trails the commanded step by ~1–2 ticks. The `backup_distance` accumulator still advances on the **commanded** step, so the budget closes on paper while the bot is still carrying backward velocity from the previous commanded ticks. Across phase boundaries the residual velocity leaks into COMMIT or the next TENSION, producing ~7 px of drift on S14.1-nav goldens and breaking the replay-determinism AC.

Nutts's debug summary (paraphrased): TENSION legitimately has backward intent when too close. `desired_back_pre` is positive, so the original §5 accumulator sees a backward-biased `desired_vel`, and the smoothing layer carries backward momentum 1–2 ticks past the budget boundary. `backup_distance` is per-phase and can reset, so repeated overshoots stack.

This is NOT a Nutts-side bug. It is a gap in the original §3 audit: the audit named "15 `b.position +=` writes" and treated them uniformly under one accumulator. The writes have **different semantic intents**, and lumping retreat writes in with chase writes violates determinism invariants that the chase writes don't touch.

### 1.3 Re-count of write sites (the original audit said "15"; the real count is larger)

Grepping `b.position` writes and reads, the actual site count in `combat_sim.gd` (post-S17.2-002) is 20 writes across `_move_brott`, `_do_combat_movement`, and `_apply_unstick_nudge`. The original spec's "~15" was a rough pass. The full audit is in §2.

## 3. Design principle

### 3.1 What smoothing is FOR

The scout-feel invariant is "nimble, not magical" — specifically, the **vector-direction of travel intent** should not flip instantaneously when a bot decides where it wants to go. The jerk complaint from HCD is about **intent reversals**, not about corrections. Concretely:

- A scout mid-COMMIT dash whose target jukes — intent rotates. Smoothing is correct.
- A scout whose chassis overlaps another bot and gets a separation kick — not intent; it's a geometry solver. Smoothing it makes overlap resolution laggy and re-introduces the pre-S15 moonwalk-adjacent class of bugs.
- A scout pinned to a wall with an unstick nudge — not intent; it's an escape solver. Smoothing it burns 1–3 ticks of the 8-tick escape window on rotation lag. S17.2-002 already established this bypass; §7 of the original spec flagged it.
- A scout in TENSION that decided it's too close and wants to back off — **this IS intent**, but the intent is **gated by a per-tick budget** (`backup_distance`). The budget math assumes commanded == realized. Smoothing breaks that assumption.

### 3.2 The retreat-budget invariant

Both TENSION-too-close retreat (#21) and RECOVERY retreat (#26) share the same code shape:

    if dist < ideal [- tolerance] and b.backup_distance < TILE_SIZE:
        var step: float = minf(spd, TILE_SIZE - b.backup_distance)
        b.position -= to_target.normalized() * step
        b.backup_distance += step

`backup_distance` is the S15 moonwalk-fix invariant: one tile of backward travel per phase entry, then lateral-only until the phase resets. The accumulator advances on the **commanded step**. Smoothing makes **realized** displacement lag commanded — the bot accumulates `backup_distance` on the commanded ledger while still carrying commanded-but-not-yet-realized backward velocity. Next tick the budget says "done, go lateral," smoothing keeps bleeding backward velocity, and the net displacement overshoots the 32 px cap by the realized-vs-commanded delta.

This is Nutts's ~7 px drift on S14.1-nav goldens. It's not a tuning issue; it's a structural invariant break. The budget math is tick-accurate-commanded or it's nothing.

### 3.3 Why retreat sharpness is a feature, not a bug

Combat retreat in TENSION/RECOVERY is a **discrete combat decision**: "this bot is inside its preferred engagement band, back off." The bot is not arcing around a pursuit target; it's ratcheting out of the tolerance zone and then pivoting to lateral. A visible arc on the back-off → lateral transition would dilute the TCR rhythm HCD already tuned in S13.2/S13.3. The feel goal is **smooth forward motion, crisp back-off/lateral transitions**. Retreat writes deserve to be sharp.

Defensive-stance retreat (#5) and kiting retreat (#7) share the same semantic (stance said "back off, you're too close"); smoothing them also breaks the stance-feel that S11/S13 tuned. Same bypass, same rationale.

### 3.4 Corrections and enforcement are never smoothed

Separation push (#11–13), pillar repel (#15), arena clamp (#14, #19), and the center-snap (#2) are not intent writes. They are solvers that close a constraint violation. Smoothing any of them would either (a) let the violation linger a tick, which is a correctness bug (overlap, out-of-bounds, pillar-tunneling), or (b) double-book against intent, because the next tick's desired_vel is recomputed from the **corrected** position and the solver's effect has already been consumed by the geometry.

### 3.5 Summary table — one-liner per category

| Category | Sites | Policy | Why |
|---|---|---|---|
| forward-chase | #1, #3, #4, #6, #9, #20, #25 | SMOOTH | Canonical scout-feel goal. Intent rotation → visible arc. |
| lateral-orbit | #8, #10, #22, #23, #24, #27 | SMOOTH | Intent writes; orbit-direction change benefits from arc. |
| retreat (stance or combat phase) | #5, #7, #21, #26 | BYPASS | Budget-gated (#21, #26) or stance-discrete (#5, #7); sharpness preserves invariants and combat feel. |
| separation | #11, #12, #13 | BYPASS | Geometry solver; lag produces overlap bugs (S15 class). |
| geometry-repel | #15 | BYPASS | Absolute set-out; must close the constraint. |
| enforcement (clamp, snap) | #2, #14, #19 | BYPASS | Boundary / degeneracy; absolute by definition. |
| unstick | #16, #17, #18 | BYPASS | S17.2-002 precedent; 8-tick window cannot afford rotation lag. |

## 4. Chosen approach

**Hybrid of A + D** (where D is "retreat writes are a distinct intent category"):

- **Option A verbatim:** TENSION and RECOVERY retreat writes (#21, #26) bypass `_smooth_velocity`, same as `_apply_unstick_nudge`. They write directly to `b.position` with the existing `backup_distance` bookkeeping intact.
- **Extension (D):** stance-driven retreats (#5 defensive, #7 kiting) also bypass — they are the same intent category (discrete "back off" decisions), and smoothing them would break the stance-feel S11/S13 tuned. The original spec did not separately call these out; this revision classes them together.

Not B (tick-accurate budget bookkeeping): viable but invasive. Would require `_smooth_velocity` to return both realized displacement AND backward-realized magnitude for the caller to charge against `backup_distance`. Adds a second return value, couples the smoothing layer to the retreat-budget invariant, and makes future budget tweaks cross two files. Fragile and harder to audit.

Not C (different smoothing shape on reversal): also viable and superficially elegant. Rejected because the retreat writes aren't "reversals of forward intent" — they are **a different intent category**. In TENSION too-close, there is no prior forward intent to reverse; the bot enters the phase already in retreat mode. C's zero-damping reversal doesn't model this cleanly and still leaves the realized-vs-commanded drift on the `backup_distance` ledger (the commanded step is what arms damping; damping only affects magnitude, not the backup accumulator drift).

Option A is the cleanest and matches S17.2-002's existing precedent (`_apply_unstick_nudge` bypass). Symmetry across bypass callers reduces future audit cost.

### 4.1 Justification against scout-feel invariants

- "Nimble, not magical": preserved. The **forward-chase** and **lateral-orbit** sites (14 of the 20+ intent-writes) are still smoothed — that's the entire scout-feel goal. COMMIT dash (#25) and TENSION in-band orbit (#23) are the two sites HCD most directly flagged in the S17 playtest ("mice running around"), and both stay smoothed.
- Finite max acceleration + finite angular velocity: preserved, applied to the smoothed categories.
- Reversal damping on flips > 120° for 2 ticks at 0.35×: applies **only to smoothed categories** (scope narrows — see §6 AC impact). A TENSION retreat → TENSION lateral transition within the same phase does NOT trigger reversal damping, because the retreat write doesn't feed `b.velocity`. This is correct: the bot isn't reversing forward motion; it's ending a bounded back-off and starting a lateral.
- Per-chassis caps (540 / 270 / 150 °/s): preserved, applied to the smoothed categories.
- Max speed unchanged: preserved.
- Constants in `combat_sim.gd`: preserved.
- Replay determinism preserved: **this is where the revision matters.** The original §5 pattern breaks determinism by routing retreat writes through smoothing. Bypassing them fixes the S14.1-nav goldens AND the new S17.2-003 determinism AC. No drift.
- All three chassis: preserved — the bypass is on write-site category, not chassis.

## 5. Revised implementation pattern

The original §5 recommendation ("collapse the multiple `b.position +=` calls into one `desired_vel` accumulator per tick per bot, then apply `_smooth_velocity` once") was too coarse. It treated every intent write as fungible. The revised pattern:

### 5.1 Two-lane write model

Each tick per bot, movement writes are partitioned into two lanes:

- **Smoothed-intent lane:** a single `desired_vel: Vector2` accumulator (unit: px/s, i.e. target velocity for this tick), summed across all forward-chase and lateral-orbit sites that fire this tick. At the end of `_move_brott`'s intent phase, the accumulator is passed through `_smooth_velocity(b, desired_vel, TICK_DELTA)` exactly once, which updates `b.velocity` and writes the realized displacement to `b.position`.
- **Direct-write lane:** retreat, separation, geometry-repel, enforcement, and unstick writes apply directly to `b.position` with their existing math. They do NOT feed `desired_vel`. They do NOT update `b.velocity`.

Ordering within a tick (for one bot):

1. **Intent resolution.** Compute the bot's intended movement per brain / stance / TCR phase. Each intent site contributes to `desired_vel` (smoothed) OR applies a direct write (retreat). Retreat writes still decrement `backup_distance` on the commanded step — unchanged from baseline.
2. **Smoothing apply.** If `desired_vel != Vector2.ZERO`, call `_smooth_velocity(b, desired_vel, TICK_DELTA)` once. This produces the realized displacement for the smoothed lane AND updates `b.velocity` for next tick.
3. **Corrections.** Separation push, pillar repel.
4. **Enforcement.** Arena clamp.
5. **Unstick.** `_check_and_handle_stuck` → `_apply_unstick_nudge` (already self-contained; bypasses smoothing per S17.2-002).

### 5.2 Velocity accounting on bypassed writes

A retreat write moves `b.position` but does NOT change `b.velocity`. This is a deliberate choice:

- If a bot is in pure retreat (TENSION too-close, first tick), `desired_vel == Vector2.ZERO` and `_smooth_velocity` is skipped. `b.velocity` is left at its previous value (likely the prior lateral or forward orbit velocity), which is fine — it naturally decays on the next smoothed tick because no forward desired_vel feeds it, and the magnitude blend will pull it toward zero (no matter, the decay is bounded by chassis `decel`).
- If a bot mixes retreat + lateral in one tick (TENSION budget exhausted: retreat step, then lateral perp), the retreat is direct-write, the lateral contributes to `desired_vel`, and the next smoothed apply advances `b.velocity` toward the lateral target. Clean.
- If a bot's retreat is followed (next tick) by COMMIT forward, `b.velocity` was left carrying the lateral/orbit vector from two ticks ago. `_smooth_velocity` then rotates that vector toward the COMMIT forward target over the angular-velocity cap. Visible arc preserved on the retreat → commit transition — the scout-feel goal — without the retreat itself being smoothed.

### 5.3 Pseudocode for the intent phase

Not actual GDScript; plain prose with 4-space indent for structure.

Inside `_do_combat_movement(b, base_spd)`:

        var desired_vel := Vector2.ZERO
        match b.combat_phase:
            TENSION:
                if dist > ideal + tolerance:
                    # forward-chase: smoothed
                    desired_vel += to_target.normalized() * b.current_speed
                elif dist < ideal - tolerance:
                    if b.backup_distance < TILE_SIZE:
                        # retreat: direct write, commanded-step budget
                        var step := minf(orbit_spd, TILE_SIZE - b.backup_distance)
                        b.position -= to_target.normalized() * step
                        b.backup_distance += step
                    else:
                        # budget exhausted: lateral-orbit, smoothed
                        desired_vel += perp * float(b.orbit_direction) * b.current_speed
                else:
                    desired_vel += perp * float(b.orbit_direction) * b.current_speed
                # drift: smoothed (it's a lateral intent nudge)
                if drift_offset != 0.0:
                    desired_vel += perp * (drift_offset / TICK_DELTA)
            COMMIT:
                if dist > commit_target_dist:
                    desired_vel += to_target.normalized() * b.current_speed
            RECOVERY:
                if dist < ideal and b.backup_distance < TILE_SIZE:
                    # retreat: direct write
                    var step := minf(recovery_spd, TILE_SIZE - b.backup_distance)
                    b.position -= to_target.normalized() * step
                    b.backup_distance += step
                else:
                    desired_vel += perp * float(b.orbit_direction) * b.current_speed
        if desired_vel != Vector2.ZERO:
            b.position += _smooth_velocity(b, desired_vel, TICK_DELTA)

`_smooth_velocity` signature unchanged from original §4.2. It still returns realized displacement and updates `b.velocity` as a side effect.

### 5.4 Migration checklist for Nutts (revised)

- Read every site in §2's table. Tag each call site with its category (forward-chase / retreat / lateral-orbit / separation / geometry-repel / enforcement / unstick).
- Forward-chase + lateral-orbit → migrate to `desired_vel` accumulator.
- Retreat → leave as-is (`b.position -= ...`). Do NOT touch `b.velocity`. Do NOT touch `backup_distance` math.
- Separation, geometry-repel, enforcement, unstick → leave as-is.
- After the intent phase, add the single `_smooth_velocity` apply. Before the separation loop.
- Verify: `b.velocity` is written exactly once per tick (inside `_smooth_velocity`) and read exactly twice (inside `_smooth_velocity` at the start, for next-tick continuity).

## 6. Impact on acceptance criteria

Relative to original spec §8. Re-scoping, not adding ACs.

- **AC-1 Visible-arc test (unchanged in shape, scoped to forward intent).** The canned scenario must use a **forward pursuit** (Scout committing, target moves behind Scout → Scout rotates toward new target position). A TENSION → COMMIT forward transition qualifies. A TENSION too-close retreat → lateral transition does NOT qualify — retreat is unsmoothed. AC wording should read "In a scenario where Scout's forward-intent direction reverses > 120°, Scout must take ≥ 3 ticks to complete the rotation."
- **AC-2 No straight-line slowdown.** Unchanged. Forward chase is still smoothed; acceleration unchanged.
- **AC-3 Pursuit still closes.** Unchanged. Regression tolerance (+15%) still applies.
- **AC-4 Phase-transition damping visible.** **Scoped to forward-intent transitions only.** A COMMIT → RECOVERY transition where RECOVERY immediately hits the retreat branch will NOT show magnitude dip (retreat bypasses smoothing). The dip IS visible when RECOVERY's retreat budget is already exhausted and the phase spawns directly into lateral (common in close-quarters). AC should read: "On a forward-intent direction reversal across phase boundary, `b.velocity.length()` must dip by ≥ 35% for 2 ticks."
- **AC-5 No teleport.** Unchanged. The bound already permits the retreat writes (they are bounded by `backup_distance` budget and `recovery_spd`).
- **AC-6 HCD subjective read.** Unchanged.
- **AC-T1 Unit test for `_smooth_velocity`.** Unchanged — the helper itself is untouched.
- **AC-T2 Replay-determinism.** **THIS IS THE DETERMINISM AC THAT BLOCKED NUTTS.** With the retreat-bypass in place, S14.1-nav goldens should reproduce byte-identical. Nutts's ~7 px drift was the signal; this revision fixes it. AC unchanged in wording, passes on the revised pattern.
- **AC-T3 Regression.** Unchanged.

**New AC recommended (AC-T4):** with a synthetic TENSION-too-close scenario (bot spawned inside `ideal - tolerance`), total backward displacement across one full TCR cycle must be ≤ `TILE_SIZE * 2 + epsilon` (one tile for TENSION retreat + one tile for RECOVERY retreat, per-phase budget). This catches the realized-vs-commanded overshoot directly. Small unit test, can live in `test_s17_2_scout_feel.gd`.

## 7. Tuning hooks

The bypass policy is structural, not tunable — there is no "smooth retreat a little" knob that would make sense. However, two developer-facing toggles are recommended for HCD feel-check playtest:

- `const DEBUG_DRAW_VELOCITY_LANE: bool` (default `false`). When true, the debug overlay draws `b.velocity` in one color (smoothed lane) and a separate arrow for the retreat direction on ticks where retreat fires. Makes the two-lane model visible. `arena_renderer.gd` dev flag.
- `const SMOOTHING_BYPASS_ENABLED: bool` (default `true`). Kill-switch that routes retreat writes back through the smoothing accumulator. Exists only for A/B playtest ("does the crisp retreat actually read better than smoothed?"). If HCD's feel-check decides smoothed retreat is acceptable, we'd still need the B-option tick-accurate budget bookkeeping to keep determinism — so the kill-switch is an investigative tool only, not a shipping knob.

No new `combat_sim.gd` constants added. No `brott_state.gd` field changes beyond what original §4.1 already specified. Scope-gate status unchanged: still `combat/**` only.

## 9. One-line summary for Riv

The original spec's "one accumulator, one smooth call" pattern lumps combat-intent retreat writes in with forward-chase writes and breaks the `backup_distance` budget invariant (S14.1-nav ~7 px drift). Fix: partition write sites into smoothed-intent (forward-chase + lateral-orbit, 14 sites) and direct-write (retreat + separation + geometry-repel + enforcement + unstick, 13 sites). Retreat writes bypass smoothing same as `_apply_unstick_nudge` does. Scout-feel goal preserved on the sites HCD cared about; determinism preserved on the sites the goldens care about. Complexity unchanged (M).
