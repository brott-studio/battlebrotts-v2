## 6. Critical invariant — preview-shown name == commit name

This is the central correctness property of the design. Verified by code inspection on `main`:

1. **`shop_screen.gd::_prepare_trick_for_modal(trick)`** deep-duplicates the trick, then for each choice's `effect_type` / `effect_type_2`, if the type is `ITEM_GRANT` / `ITEM_LOSE` and the `effect_value` is a `random_*` pool token, it calls `ItemTokens.resolve_token(tok)` **once** and writes the resulting concrete token back to `c[field_v]`. After this step, the patched dict contains only DIRECT tokens for all item effects.
2. **`trick_choice_modal.show_trick(patched)`** → `_populate_previews(patched)` → `_preview_for_choice(choice)` reads `choice["effect_value"]` (now concrete) and calls `_name_from_token`, which calls `ItemTokens.resolve_token` on a DIRECT token. `item_tokens.gd::resolve_token` (lines 52–70): DIRECT tokens take the `DIRECT.has(token)` branch with **no RNG call**; `display_name` is a pure lookup in `WeaponData` / `ArmorData` / `ModuleData`. **No RNG divergence possible.**
3. **On A/B commit**, `shop_screen` calls `game_state.apply_trick_choice(patched, choice_key)` on the **same patched dict** — the same concrete token the preview used. The item granted/lost is therefore byte-identical to the previewed name.
4. **On skip**, the patched dict is discarded, `apply_trick_choice` is not called, no mutation occurs.

**The invariant holds structurally.** No fixture-level sanity check is needed because the only source of non-determinism (the `random_*` pool resolution) is consumed exactly once per trick-visit and cached on the patched dict before both the preview and the commit read from it.

## 7. ESC parity confirmation

- The same `_resolving` one-shot guard is shared between the Skip button-press path and the ESC path — cannot double-fire.
- ESC is ignored until `_overlay.modulate.a >= 0.95` (fade-in complete), preventing input-race during the 0.2s fade-in tween.
